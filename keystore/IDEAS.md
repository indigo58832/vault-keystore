## 1. Автоудаление при блокировке

**Что**: если при проверке ключ возвращает код блокировки (например `0xC004C006` или подобный) — удалить его автоматически.

**Зачем**: 006-коды значат что ключ битый/невалидный, в складе он не нужен и только засоряет.

**ВАЖНОЕ исключение — MAK ключи НЕ удалять**:
- Если MAK ушёл в 006 / исчерпан / заблокирован → **переместить в категорию «Склад старых ключей»**, а не удалить.
- Причина: MAK ключи могут потом понадобиться (история, recode-история, проверка спорных случаев).

**Какие коды считать "блок → удалить"** (для Retail/OEM):
- `0xC004C003` — заблокирован MS
- `0xC004C060` — отозван
- `0xC004C006` — пользователь упомянул конкретно его
- (нужно уточнить полный список, какие реально означают "выкидывать")

**Какие коды НЕ удалять**:
- `0xC004C008`, `0xC004C020` — это "телефон" (онлайн вышел, phone работает) → оставить
- Всё что ✓ valid → оставить

## 2. Автопроверка при добавлении

**Что**: занёс ключ через «+ Добавить ключи» → сразу автопроверка (статус, тип, MAK count).

**КРИТИЧНЫЙ нюанс — для Retail/OEM проверка тратит дорогую активацию**:
- **MAK ключи**: проверять автоматически можно (запрос BatchActivation не списывает активацию).
- **Retail / OEM Online**: проверять автоматически **НЕЛЬЗЯ** — Consume у этих ключей съест единственную активацию, а они дорогие.

**Решение**:
- При автопроверке делать **только без Consume** (query + MAK count).
  - Это даст тип/эдишн/online-valid статус.
  - Для MAK дополнительно даст счётчик активаций.
  - Для Retail/OEM скажет "Online-valid" — без расхода.
- Точные коды ошибок (через Consume) — только **по явной кнопке** «Проверить с расходом», когда пользователь сам захотел.

**Где переключать**: в настройках программы галочка «Автопроверка при добавлении» (по дефолту on), и отдельная галочка «Использовать Consume при автопроверке» (по дефолту off — берегли активации).

## 3. Умное поведение Quick Check (горячая клавиша Ctrl+Shift+S)

**Сейчас**: окно проверки просто показывает результат и не трогает базу Vault.

**Хочу**: интеграция с базой.

### Сценарии:

**A. Ключ ЕСТЬ в нашем пуле**:
- Quick Check проверил
- Если ушёл в код блокировки (006 и т.п.) → **удалить из базы**
  - Если ключ MAK → переместить в «Склад старых ключей» вместо удаления
- Если ✓ valid / телефон → обновить статус в базе (как при «Проверить выделенные»)

**B. Ключ НЕ в нашем пуле (чужой/новый)**:
- Quick Check проверил
- Если ✓ valid → **добавить в пул** (в соответствующую категорию по типу/эдишн)
  - Какая категория — определяется автоматически по результату: например для `Office24_ProPlus2024VL_MAK_AE2` → категория «Office 24 PP Online» или «Office 24 PP Phone»? Тут нужен маппинг (description → category_name).
  - Если непонятно куда — спросить или положить в категорию «Inbox / без категории».
- Если заблокирован / битый → не добавлять.

### Маппинг description → category

Нужно правило соответствия (когда ключ "норм" — куда его класть). Пример:
```
Win 10 RTM Professional Volume:MAK     → Windows Pro Phone (или Online — по типу активации?)
Win 10 RTM Core Volume:MAK             → Windows Home phone
Office24_ProPlus2024VL_MAK_AE*         → Office 24 PP Online или Office 24 PP Phone (нужно решить)
Office16_ProPlusMSDNR_Retail           → Office PP 16 Phone (?)
...
```

**Уточнить у пользователя**: по каким критериям различать Online vs Phone категорию при автодобавлении (по умолчанию класть в Phone, потому что онлайн-активаций мало?).

## 4. Автозамена мёртвого ключа при Quick Check

**Сценарий**:
- Покупатель пишет «ключ не работает»
- Я выделяю его ключ в чате
- Ctrl+Shift+S → Quick Check
- Программа проверила: ключ **мёртв** (заблокирован/код 006/060)
- **Программа автоматически находит замену в нашем пуле**: ключ того же продукта/эдишена со статусом «свободен» или «телефон»
- Перед показом замены проверяет что она **живая** (online query без расхода)
- Показывает её в окне → можно скопировать и отправить покупателю

**Если ключ жив или phone (008/020)**:
- Замена не нужна
- Просто показать результат проверки как сейчас

**Как искать «аналогичную» замену**:
- По полю `description` или `edition_id` из проверки. Например, мёртвый ключ был `Win 10 RTM Professional Retail` → ищем в категориях нашего пула ключи с тем же description.
- Альтернатива: маппинг description → category (как в идее №3) — берём первый available/phone_only ключ из этой категории.

**Что показывать в UI Quick Check**:
- Результат проверки исходного ключа (как сейчас)
- Блок «✓ Найдена замена: XXXXX-... (Windows Home phone, осталось 12 шт.)»
- Кнопка «Скопировать замену»
- Опционально: кнопка «Скопировать и пометить как продано»

**Если замены нет**:
- «✗ Нет подходящих ключей в пуле для этого продукта»
- Можно показать в каких категориях вообще нет товара (полезно для планирования закупки).

## 7. Автоочистка / периодическая проверка (background)

**Что**: Vault раз в N часов фоном проверяет весь склад на серверах MS и обновляет статусы. Мёртвые автоматически удаляются (или MAK уезжают в архив). Это автоматизация того что сейчас делается вручную кнопкой «Проверить все».

**Конкретика**:
- По расписанию (`QTimer` в `app.py`, не в UI потоке): раз в 4 часа.
- Запускается только когда сервер `:17777` живой и Vault поднят (если выключил программу — следующий запуск с задержкой).
- **БЕЗ Consume по дефолту** — независимо от категории. Это безопасно: мы только читаем статусы.
- MAK Count для Volume:MAK обновляется — это не списывает активацию.
- Дроссель: между запросами 1-2 сек, чтобы не было burst. Если 200 ключей × 2 сек = ~7 мин — это нормально для фона.
- Можно задать в настройках: интервал (1ч / 4ч / 12ч / сутки / выкл), брать ли Consume для Phone-категорий (по умолчанию выкл).

**Безопасность** (риски запросов к Microsoft):
- Endpoint `licensing.microsoft.com` и `activation.sls.microsoft.com` — публичные, миллиарды запросов в день от Windows-ПК во всём мире. От нас 100-500 запросов в день — капля.
- НИКОГДА не делать burst (много запросов за секунду) — нужен sleep между ними.
- НЕ использовать Consume периодически на Retail — это списывание активаций, это **видный паттерн** для MS, могут забанить ключ/IP.
- Дроссель 1.5-2 сек между запросами.
- Раз в 4 часа — норма. Раз в 5 минут на каждый ключ — уже подозрительно.

**Опциональное расширение**:
- Перед каждой автопроверкой пинговать `/health` сервера: если выключен — не пробовать.
- Логировать в файл `~/.local/share/keystore/auto_check.log` что когда проверилось и что изменилось.

### Правило по категориям для автопроверки

**ЯВНОЕ РЕШЕНИЕ пользователя**: Online-ключи автоматически НЕ ПРОВЕРЯТЬ. Они дорогие
(условно купил 100 шт за $50, хочешь продавать по $10), и если автопроверка
ошибочно сочтёт их «мёртвыми» — программа удалит и пользователь не заметит
без записи. Финансовый риск > выгода от автоматизации. Online проверяются только
**вручную** через кнопку «Проверить» в Vault или при жалобе покупателя через Quick Check.

**УТОЧНЕНО 2026-05-13**: правило теперь per-key (по `is_mak`), а не только по имени категории.
Воркер просыпается раз в 6 ч, но каждый ключ решает сам, надо ли его сейчас чекать.

| Состояние ключа | Phone-категория | Online-категория |
|---|---|---|
| `last_checked_at = NULL` | пропуск (тип неизвестен) | пропуск |
| `is_mak = True` | **раз в 24 ч** (consume жжёт 1 актив. из ~1000) | **раз в 24 ч** |
| `is_mak = False` (Retail/OEM) | **раз в 6 ч** (consume бесплатен — phone-only ключ сам отбьёт 008/020) | **НИКОГДА** ($3-8 в трубу) |
| `status = sold` | пропуск (история) | пропуск |
| Архив | пропуск | пропуск |

Экономика на одну проверку:
- MAK consume: ~1-4 цента (1/1000 активаций × $20-40 за ключ).
- Phone-only Retail/OEM consume: бесплатно (сервер отбивает 008/020 без расхода).
- **Online Retail/OEM consume**: $3-8 за раз (вся стоимость ключа). **Поэтому никогда автоматом.**

Реализация: `keystore/auto_check.py::_needs_check()`. Online = категория без 'phone' в имени.

## 8. Авто-сортировка ключей по категориям при импорте

**Что**: новая кнопка «Импорт без категории» или «Авто-разложить». Открывает окно, в которое можно вставить любой бардак с ключами разных продуктов. Программа сама раскладывает по категориям.

**Логика**:
1. Извлекаем все ключи regex'ом из вставленного текста (как сейчас в `AddKeysDialog`).
2. Для каждого ключа: проверяем через сервер (только query, без Consume — у нас же ещё нет категории, не знаем разрешать ли расход).
3. По результату (`description`, `is_mak`) → `policy.category_for_description` → имя категории.
4. Если категория есть в БД — кладём туда. Если нет — создаём.
5. В конце отчёт: «Из 47 ключей разложено по 5 категориям: Win10ProPhone=12, Office24PPPhone=20, ...; не определилось 3 — в Inbox».

**Прогресс**: показывать прогресс-бар, проверка через MS не моментальная.

**UI**: можно вместо отдельной кнопки сделать как опцию в диалоге «Добавить ключи»: чекбокс «Авто-сортировать (проверка через MS, медленнее)».

### 6. Парсер бесплатных ключей в интернете
**Что**: отдельный проект — бот/парсер который сканит публичные источники (форумы, телеграм-каналы, паблики, обзоры активаторов и т.п.) и находит свежие public-leaked ключи Microsoft.

**Зачем**: получать бесплатный товар автоматически — кладётся в склад Vault на проверку.

**Интеграция с Vault** (когда парсер будет готов):
- Парсер пишет найденные ключи в свою БД или JSON-файл (или прямо в SQLite Vault через общий путь)
- Vault имеет специальную категорию «Inbox / из интернета»
- При появлении новых ключей — авто-проверка через сервер `:17777` (без consume — это чужие, мы их не покупали, не знаем что в них)
- Живые после проверки автоматически перекладываются в обычные категории по `policy.category_for_description`
- Мёртвые — удаляются

**Архитектурно**:
- Парсер пишет в файл-«дроп» (`~/.local/share/keystore/inbox/keys.txt` или подобное)
- В Vault watcher читает этот файл периодически, переносит в БД и проверяет

**Когда делать**: после того как закончу парсер.
